# 09 — Roadmap produit

## 1. Finalité

La roadmap produit traduit le Blueprint en séquence d'exécution réaliste.

Elle ne doit pas être lue comme une liste de fonctionnalités à livrer. Elle organise la montée en puissance de Mbàmbulaan autour de preuves successives :

- preuve d'utilité terrain ;
- preuve de coordination ;
- preuve d'adoption ;
- preuve de valeur institutionnelle ;
- preuve de monétisation ;
- preuve de réplication.

Chaque étape doit réduire un risque majeur avant d'engager davantage de ressources.

## 2. Principe de progression

Mbàmbulaan ne doit pas chercher à construire dès le départ l'infrastructure nationale complète.

La progression recommandée est :

```text
Problème ciblé
   ↓
Territoire pilote
   ↓
Flux de coordination prouvé
   ↓
Produit opérable
   ↓
Modèle économique validé
   ↓
Réplication multi-territoires
   ↓
Infrastructure nationale
```

## 3. Les risques à réduire dans l'ordre

La roadmap doit traiter les risques dans cet ordre :

1. risque de non-utilité ;
2. risque de non-adoption ;
3. risque de non-coordination ;
4. risque de mauvaise qualité des données ;
5. risque de dépendance à un sponsor unique ;
6. risque de modèle économique non viable ;
7. risque de complexité excessive ;
8. risque de non-réplication.

Construire plus vite n'a aucun intérêt si ces risques restent ouverts.

## 4. Phase 0 — Cadrage et preuve de besoin

### Objectif

Vérifier qu'un problème de coordination prioritaire mérite une réponse Mbàmbulaan.

### Travaux

- choisir un territoire pilote ;
- identifier une transformation prioritaire ;
- cartographier les acteurs ;
- documenter les flux actuels ;
- mesurer les délais, pertes et doublons ;
- identifier les décideurs et payeurs potentiels ;
- définir les critères de succès du pilote ;
- sécuriser un sponsor opérationnel local ;
- formaliser la gouvernance minimale.

### Livrables

- diagnostic du territoire ;
- carte des acteurs ;
- parcours actuel ;
- problème prioritaire ;
- hypothèses de valeur ;
- protocole pilote ;
- indicateurs de référence.

### Critères de passage

- le problème est réel et fréquent ;
- plusieurs acteurs reconnaissent la friction ;
- un acteur local accepte de participer ;
- un décideur institutionnel s'intéresse au résultat ;
- la valeur attendue peut être mesurée ;
- le périmètre est suffisamment étroit pour être testé.

### Ce qui ne doit pas être construit

- marketplace ;
- système national ;
- intégrations lourdes ;
- moteur de recommandation avancé ;
- paiements ;
- application complète pour tous les acteurs.

## 5. Phase 1 — Pilote opérationnel minimum

### Objectif

Prouver qu'un flux de coordination concret fonctionne de bout en bout.

### Flux de référence

```text
Signalement
   ↓
Qualification
   ↓
Coordination
   ↓
Action
   ↓
Suivi
   ↓
Retour au contributeur
```

### Capacités prioritaires

- identité et organisation simples ;
- référentiel territorial ;
- signalement ;
- qualification ;
- affectation et responsabilités ;
- suivi d'actions ;
- notifications ;
- retour au contributeur ;
- historique minimal ;
- gestion documentaire légère.

### Acteurs prioritaires

- acteur terrain ;
- relais local ;
- CLPA ou collectivité ;
- opérateur Mbàmbulaan ;
- sponsor institutionnel en consultation.

### Mode opératoire

Le pilote peut combiner produit numérique et opérations manuelles.

L'objectif n'est pas d'automatiser immédiatement toutes les étapes, mais de prouver le flux métier, la valeur et les responsabilités.

### Critères de succès

- acteurs réellement actifs ;
- signalements pris en charge ;
- réduction du délai de coordination ;
- retours envoyés aux contributeurs ;
- décisions ou actions documentées ;
- sponsor capable d'expliquer la valeur créée ;
- volonté explicite de poursuivre.

### Décision de fin de phase

- poursuivre ;
- ajuster le problème ciblé ;
- changer de territoire ;
- arrêter si aucune valeur opérationnelle n'est démontrée.

## 6. Phase 2 — Produit opérable 1.0

### Objectif

Transformer le pilote en produit stable, réutilisable et administrable.

### Capacités ajoutées

- gestion structurée des organisations et rôles ;
- droits d'accès ;
- jumeau territorial simplifié ;
- gestion des initiatives ;
- risques et jalons ;
- tableaux de travail par responsabilité ;
- recherche ;
- audit ;
- imports et exports ;
- administration locale ;
- qualité de données ;
- capitalisation des apprentissages.

### Exigences structurantes

- mobile-first ;
- faible connectivité ;
- simplicité ;
- traçabilité ;
- sécurité ;
- capacité d'administration sans équipe technique ;
- coût d'exploitation maîtrisé.

### Critères de passage

- le produit fonctionne sans accompagnement quotidien des fondateurs ;
- les relais locaux peuvent administrer les usages courants ;
- les données principales sont suffisamment fiables ;
- le flux principal est reproductible ;
- les coûts opérationnels sont connus ;
- un premier contrat, financement ou engagement de continuité est obtenu.

## 7. Phase 3 — Validation du modèle économique

### Objectif

Démontrer qu'un ou plusieurs acteurs paient durablement pour la valeur créée.

### Hypothèses prioritaires à tester

- abonnement institutionnel pour coordination territoriale ;
- contrat programme ou licence publique ;
- abonnement bailleur ou ONG pour suivi de portefeuille ;
- frais de déploiement et intégration ;
- services de structuration d'initiatives ;
- offre premium d'analyse et de pilotage.

### Tests à conduire

- volonté de payer ;
- cycle de vente ;
- budget disponible ;
- niveau de service attendu ;
- coût de déploiement ;
- coût de support ;
- marge brute ;
- dépendance au sur-mesure ;
- taux de renouvellement attendu.

### Critères de succès

- au moins un payeur réel ;
- contrat ou engagement pluriannuel privilégié ;
- marge positive sur le périmètre vendu ;
- part de produit standard supérieure au spécifique ;
- coût de déploiement décroissant ;
- proposition de valeur compréhensible sans longue démonstration.

### Alerte

Un financement de projet n'est pas encore une validation du modèle économique s'il n'existe aucune perspective de continuité après le projet.

## 8. Phase 4 — Réplication multi-territoires

### Objectif

Prouver que Mbàmbulaan peut être déployé dans plusieurs territoires sans reconstruction complète.

### Capacités ajoutées

- modèles de territoire ;
- référentiels partagés ;
- configuration locale ;
- comparaison interterritoriale ;
- portefeuilles de transformation ;
- consolidation régionale et nationale ;
- communautés de pratique ;
- réplication des initiatives ;
- intégrations standards.

### Critères de succès

- deuxième territoire déployé plus vite que le premier ;
- majorité du produit réutilisée ;
- rôles et flux configurables ;
- indicateurs comparables ;
- administration locale réelle ;
- coût marginal de déploiement en baisse ;
- valeur démontrée dans des contextes différents.

## 9. Phase 5 — Infrastructure sectorielle

### Objectif

Faire de Mbàmbulaan une infrastructure de coordination reconnue à l'échelle de la filière.

### Capacités avancées

- intégrations institutionnelles ;
- portefeuilles nationaux ;
- gouvernance multi-acteurs ;
- mesure d'impact consolidée ;
- coordination des financements ;
- moteur de recommandation ;
- réplication assistée ;
- alertes prédictives ;
- accès contrôlé à des partenaires financiers ;
- API écosystème.

### Conditions nécessaires

- gouvernance crédible ;
- modèle économique diversifié ;
- forte qualité des données ;
- valeur démontrée pour les acteurs terrain ;
- confiance institutionnelle ;
- capacité technique et opérationnelle ;
- indépendance suffisante vis-à-vis d'un financeur unique.

## 12. Ce qui doit rester hors roadmap initiale

Les éléments suivants ne doivent entrer dans la roadmap qu'après preuve de valeur :

- marketplace généraliste ;
- paiement intégré ;
- crédit en propre ;
- gestion comptable ;
- traçabilité exhaustive de toute la chaîne ;
- intelligence artificielle généralisée ;
- application native complexe ;
- couverture nationale immédiate ;
- personnalisation lourde pour chaque institution ;
- collecte massive de données sans usage clair.

## 13. Méthode de priorisation

Chaque demande doit être évaluée selon huit critères :

1. impact sur la coordination ;
2. valeur métier ;
3. fréquence du besoin ;
4. criticité ;
5. nombre d'acteurs concernés ;
6. potentiel de revenu ;
7. effort et dépendances ;
8. contribution à l'avantage concurrentiel.

Une fonctionnalité intéressante mais sans amélioration mesurable de la coordination ne doit pas devenir prioritaire.

## 14. Gouvernance de roadmap

La roadmap doit être revue régulièrement par un comité produit restreint réunissant :

- produit ;
- opérations terrain ;
- technologie ;
- business ;
- représentant des utilisateurs ;
- sponsor institutionnel lorsque pertinent.

Chaque arbitrage doit préciser :

- l'hypothèse testée ;
- la preuve attendue ;
- le coût ;
- le responsable ;
- le délai ;
- la décision possible à l'issue.

## 15. Indicateurs de progression

### Utilité

- problèmes réellement résolus ;
- temps économisé ;
- décisions améliorées ;
- actions déclenchées.

### Adoption

- acteurs actifs ;
- fréquence d'usage ;
- taux de retour ;
- qualité des contributions.

### Coordination

- délais ;
- engagements tenus ;
- doublons évités ;
- blocages résolus.

### Économie

- revenus ;
- marge brute ;
- coût par territoire ;
- coût de support ;
- renouvellement ;
- concentration client.

### Réplication

- délai de déploiement ;
- part de configuration ;
- part de produit standard ;
- coût marginal ;
- réutilisation des connaissances.

## 16. Principes de décision

- ne pas automatiser un processus que nous ne comprenons pas ;
- ne pas généraliser avant d'avoir prouvé localement ;
- ne pas confondre activité et impact ;
- ne pas accepter un développement spécifique qui déforme le cœur produit ;
- ne pas dépendre durablement d'un financeur unique ;
- ne pas collecter une donnée sans usage, responsable et règle de gouvernance ;
- ne pas passer à l'échelle avant d'avoir un modèle opérable.

## 17. Principe directeur

> La roadmap de Mbàmbulaan doit financer l'apprentissage avant de financer la complexité, puis financer la réplication avant de financer l'échelle.
