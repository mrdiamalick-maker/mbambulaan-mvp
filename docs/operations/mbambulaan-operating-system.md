# Mbàmbulaan Operating System

## 1. Finalité

Le Mbàmbulaan Operating System décrit comment l’entreprise transforme une difficulté terrain en valeur économique, institutionnelle et opérationnelle.

Ce document ne décrit pas seulement le produit. Il décrit le système de fonctionnement de Mbàmbulaan comme entreprise :

- comment un territoire est ouvert ;
- comment un pilote est cadré ;
- comment le produit est adapté sans dériver ;
- comment les opérations sont exécutées ;
- comment les résultats sont mesurés ;
- comment un pilote devient un contrat ;
- comment un contrat devient un revenu récurrent ;
- comment l’entreprise se déploie sans dépendre entièrement des fondateurs.

---

# 2. Principe directeur

Mbàmbulaan doit rester une entreprise produit fondée sur une infrastructure de coordination.

Le service, l’accompagnement et le terrain sont nécessaires, mais ils doivent servir trois objectifs :

1. accélérer l’adoption ;
2. améliorer le produit ;
3. convertir l’usage en revenu récurrent.

Ils ne doivent pas transformer l’entreprise en cabinet de conseil ou en société de prestations sur mesure.

---

# 3. Moteur de valeur central

Le moteur central est le suivant :

> Une difficulté est remontée, comprise, attribuée, suivie, résolue et capitalisée.

Chaque nouveau module, partenariat ou source de revenu doit renforcer ce moteur.

Une fonctionnalité n’est prioritaire que si elle améliore au moins un des éléments suivants :

- qualité de l’information ;
- clarté des responsabilités ;
- vitesse de prise en charge ;
- capacité de coordination ;
- visibilité sur les résultats ;
- valeur pour un payeur.

# 4. Chaîne de valeur opérationnelle

## Étape 1 — Détection d’une opportunité

Une opportunité peut venir :

- d’un ministère ;
- d’une direction technique ;
- d’un programme ;
- d’une collectivité ;
- d’un CLPA ;
- d’une organisation professionnelle ;
- d’un opérateur privé ;
- d’un bailleur.

Critères minimum :

- problème concret ;
- acteur responsable identifiable ;
- territoire ou population définis ;
- accès possible au terrain ;
- intérêt potentiel d’un payeur.

## Étape 2 — Qualification

Questions obligatoires :

- quel problème est réellement mal coordonné ?
- qui souffre du problème ?
- qui décide ?
- qui peut agir ?
- qui paie aujourd’hui le coût du dysfonctionnement ?
- qui pourrait payer pour améliorer la situation ?
- quelles données sont déjà disponibles ?
- quels dispositifs existent déjà ?

Décision :

- poursuivre ;
- différer ;
- refuser.

## Étape 3 — Proposition de pilote

Le pilote doit rester limité :

- un territoire ;
- un moteur métier principal ;
- un nombre réduit d’acteurs ;
- une durée courte ;
- des indicateurs mesurables ;
- une décision de sortie explicite.

## Étape 4 — Co-construction terrain

Objectifs :

- comprendre les pratiques réelles ;
- valider le vocabulaire ;
- identifier les responsabilités ;
- éliminer les étapes inutiles ;
- vérifier la valeur pour chaque acteur.

Sorties attendues :

- parcours cible ;
- acteurs et rôles ;
- cas d’usage ;
- données minimales ;
- protocole de mesure ;
- plan d’accompagnement.

## Étape 5 — Configuration et déploiement

Le produit doit être configuré, non réécrit.

Les adaptations autorisées :

- vocabulaire ;
- rôles ;
- règles de visibilité ;
- statuts ;
- indicateurs ;
- notifications ;
- modèles de rapports.

Les adaptations interdites sans arbitrage produit :

- développement spécifique non réutilisable ;
- logique métier propre à un seul client ;
- contournement de l’architecture ;
- création d’un second produit parallèle.

## Étape 6 — Exploitation du pilote

Rituels minimum :

- revue opérationnelle hebdomadaire ;
- suivi des blocages ;
- revue produit courte ;
- suivi des indicateurs ;
- journal des apprentissages.

## Étape 7 — Restitution

La restitution doit répondre à cinq questions :

1. le problème a-t-il été mieux coordonné ?
2. les acteurs ont-ils utilisé le dispositif ?
3. les délais ont-ils été améliorés ?
4. les résultats sont-ils visibles ?
5. un payeur souhaite-t-il poursuivre ?

## Étape 8 — Conversion commerciale

Options :

- abonnement territoire ;
- licence programme ;
- extension multi-sites ;
- contrat annuel ;
- nouveau pilote financé ;
- intégration à un dispositif existant.

Un pilote sans plan de conversion est incomplet.

# 5. Cycle de vie d’un territoire

## Niveau 0 — Prospection

État : aucun engagement.

Sortie attendue : interlocuteur qualifié et problème prioritaire.

## Niveau 1 — Pré-cadrage

État : intérêt confirmé.

Sortie attendue : sponsor, point focal, périmètre potentiel.

## Niveau 2 — Cadrage

État : pilote défini.

Sortie attendue : objectifs, acteurs, indicateurs, budget et gouvernance.

## Niveau 3 — Préparation

État : produit configuré et acteurs préparés.

Sortie attendue : lancement prêt.

## Niveau 4 — Pilote actif

État : usage réel.

Sortie attendue : données d’usage et premiers résultats.

## Niveau 5 — Validation

État : preuve de valeur documentée.

Sortie attendue : décision de poursuivre, adapter ou arrêter.

## Niveau 6 — Contrat récurrent

État : relation économique durable.

Sortie attendue : revenu annuel, gouvernance stabilisée, déploiement standardisé.

## Niveau 7 — Réplication

État : extension à plusieurs sites ou cas d’usage.

Sortie attendue : modèle déployable avec dépendance réduite aux fondateurs.

# 6. Gates de décision

## Gate 1 — Problème valide

Conditions :

- problème concret ;
- coût ou impact identifiable ;
- acteurs concernés accessibles ;
- solution actuelle insuffisante.

## Gate 2 — Sponsor valide

Conditions :

- décideur ou prescripteur identifié ;
- point focal disponible ;
- accès terrain possible ;
- intérêt institutionnel confirmé.

## Gate 3 — Pilote viable

Conditions :

- périmètre limité ;
- indicateurs ;
- budget ;
- responsabilités ;
- protocole de données.

## Gate 4 — Produit prêt

Conditions :

- parcours compréhensible ;
- configuration terminée ;
- tests effectués ;
- support prêt ;
- sécurité minimale validée.

## Gate 5 — Usage réel

Conditions :

- acteurs actifs ;
- situations créées ;
- coordination observée ;
- données exploitables.

## Gate 6 — Valeur démontrée

Conditions :

- résultats mesurables ;
- amélioration perçue ;
- sponsor satisfait ;
- apprentissages documentés.

## Gate 7 — Conversion économique

Conditions :

- payeur identifié ;
- offre proposée ;
- budget disponible ;
- décision de poursuite.

# 10. Règles de priorisation produit

Chaque demande est évaluée selon huit critères :

1. améliore-t-elle la coordination ?
2. crée-t-elle une valeur métier claire ?
3. combien d’acteurs en bénéficient ?
4. un payeur la valorise-t-il ?
5. est-elle réutilisable ?
6. réduit-elle le coût de déploiement ?
7. crée-t-elle un avantage concurrentiel ?
8. est-elle nécessaire maintenant ?

## Score de décision

- 0 : aucun effet ;
- 1 : effet faible ;
- 2 : effet moyen ;
- 3 : effet fort.

Une demande spécifique avec faible réutilisabilité et sans payeur est rejetée ou facturée comme exception clairement isolée.

---

# 11. Anti-dérive services

## Signaux d’alerte

- plus de 30 % du temps produit consacré à un seul client ;
- fonctionnalités non réutilisables ;
- facturation principalement en jours-homme ;
- dépendance à des ateliers permanents ;
- faible usage autonome ;
- absence de contrat récurrent ;
- coût de déploiement qui augmente avec chaque site.

## Réponses

- standardiser ;
- configurer au lieu de développer ;
- créer des playbooks ;
- former des relais ;
- imposer des limites contractuelles ;
- faire payer les exceptions ;
- arrêter les projets non convertibles.

# 16. Seuils de passage à l’échelle

Un territoire ne doit pas être répliqué tant que :

- le parcours n’est pas compris ;
- l’usage dépend fortement de l’équipe Mbàmbulaan ;
- la valeur n’est pas démontrée ;
- aucun payeur n’est identifié ;
- le coût de déploiement est inconnu ;
- les données ne sont pas fiables ;
- la gouvernance n’est pas claire.

Le passage à l’échelle est autorisé si :

- au moins un contrat est signé ;
- la marge est plausible ;
- le playbook est documenté ;
- un relais local existe ;
- le produit est réutilisable ;
- les indicateurs sont stables.

---

# 17. Gestion des exceptions

Toute demande hors standard doit être classée :

- configuration ;
- amélioration produit ;
- intégration ;
- développement spécifique ;
- prestation de conseil.

Règles :

- configuration : autorisée ;
- amélioration produit réutilisable : priorisable ;
- intégration : cadrée et facturée ;
- spécifique non réutilisable : exception payante ;
- conseil pur : limité et séparé du produit.

# 21. Résultat attendu à 24 mois

Mbàmbulaan doit être capable de :

- ouvrir un nouveau site avec un processus standard ;
- déployer un pilote en quelques semaines ;
- mesurer la valeur ;
- convertir une partie des pilotes en contrats ;
- fonctionner avec des relais locaux ;
- produire des revenus récurrents ;
- maintenir une architecture produit cohérente ;
- démontrer un avantage concurrentiel fondé sur la coordination, les données opérationnelles et la connaissance du terrain.

---

# Conclusion

Le Mbàmbulaan Operating System protège l’entreprise contre deux risques majeurs : construire un produit sans marché, ou accumuler des prestations sans produit réplicable.

La discipline centrale est la suivante :

> chaque pilote doit améliorer le produit, produire une preuve de valeur et ouvrir un chemin vers un revenu récurrent.
