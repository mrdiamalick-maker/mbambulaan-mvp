# 10 — Principes fondateurs

## 1. Finalité

Les principes fondateurs constituent la constitution produit de Mbàmbulaan.

Ils servent à arbitrer les décisions lorsque plusieurs options sont possibles, lorsque des demandes contradictoires apparaissent ou lorsque la pression commerciale pousse à sortir du cœur du produit.

Ces principes doivent rester stables dans le temps. Ils peuvent être précisés, mais ne doivent pas être contournés sans décision explicite de gouvernance.

## 2. Mbàmbulaan est une infrastructure de coordination

Mbàmbulaan n'est pas d'abord :

- une application ;
- une marketplace ;
- un ERP ;
- un tableau de bord ;
- un réseau social ;
- un outil de reporting.

Mbàmbulaan organise les relations, les responsabilités, les flux d'information, les décisions et le suivi entre les acteurs de la filière.

Toute fonctionnalité doit renforcer au moins un des éléments suivants :

- coordination ;
- confiance ;
- qualité de l'information ;
- vitesse de décision ;
- exécution ;
- mesure ;
- réplication.

## 3. Partir du problème métier, jamais de l'écran

Nous raisonnons dans cet ordre :

```text
Problème
   ↓
Acteurs
   ↓
Décisions
   ↓
Flux de valeur
   ↓
Capacités
   ↓
Moteurs métier
   ↓
Parcours
   ↓
Interfaces
```

Un écran n'est jamais une justification produit.

## 4. Chaque contribution doit produire un retour de valeur

Un acteur ne doit pas être réduit à un fournisseur de données.

Lorsqu'il contribue, il doit recevoir un bénéfice compréhensible :

- confirmation ;
- visibilité sur la prise en charge ;
- information utile ;
- décision ;
- accès à une opportunité ;
- réduction d'une friction ;
- reconnaissance de sa contribution.

Sans retour de valeur, l'adoption sera artificielle et fragile.

## 5. La donnée est un moyen, pas le produit

Mbàmbulaan ne doit pas collecter des données parce qu'elles pourraient être utiles un jour.

Toute donnée doit avoir :

- une finalité ;
- un responsable ;
- un propriétaire métier ;
- un niveau de qualité attendu ;
- une règle d'accès ;
- une durée de conservation ;
- un usage décisionnel ou opérationnel identifiable.

## 6. La confiance est une capacité fonctionnelle

La confiance ne se limite pas à la cybersécurité.

Elle repose sur :

- des identités fiables ;
- des rôles clairs ;
- la provenance des informations ;
- des validations explicites ;
- la traçabilité ;
- la gestion des consentements ;
- la possibilité de contester ;
- l'explicabilité des décisions ;
- la protection des informations sensibles.

## 7. Le terrain prime sur l'hypothèse

Une hypothèse produit n'a de valeur que si elle est confrontée aux usages réels.

Nous privilégions :

- l'observation ;
- les entretiens ;
- les pilotes ;
- les preuves ;
- les mesures ;
- les décisions fondées sur les résultats.

L'intuition des fondateurs est utile, mais elle ne remplace pas la réalité du terrain.

## 8. Le mobile et la faible connectivité sont des contraintes de base

Le produit doit être conçu pour les conditions réelles d'usage :

- smartphone ;
- bande passante limitée ;
- périodes hors ligne ;
- faible disponibilité ;
- compétences numériques variables ;
- assistance possible par un relais local.

La simplicité n'est pas une option de design. C'est une condition d'adoption.

## 9. Nous ne centralisons pas ce qui doit rester distribué

Mbàmbulaan doit permettre une gouvernance distribuée.

Les territoires, organisations et acteurs doivent pouvoir administrer les éléments relevant de leur mandat, dans un cadre commun et traçable.

L'équipe centrale ne doit pas devenir le goulot d'étranglement de chaque validation, correction ou décision.

## 10. Une donnée, un propriétaire métier

Chaque objet structurant doit appartenir à un moteur métier clairement identifié.

Les autres composants peuvent le consulter, l'enrichir ou le référencer, mais ne doivent pas dupliquer ses règles.

Ce principe protège :

- la cohérence ;
- la qualité ;
- la maintenabilité ;
- la traçabilité ;
- l'évolutivité.

## 11. L'orchestration avant la substitution

Lorsqu'un système spécialisé existe déjà et fonctionne correctement, Mbàmbulaan doit chercher à s'y connecter plutôt qu'à le reconstruire.

Mbàmbulaan doit orchestrer :

- systèmes publics ;
- outils des bailleurs ;
- solutions financières ;
- services météo ;
- plateformes statistiques ;
- outils de traçabilité ;
- référentiels externes.

Nous ne devons pas devenir un logiciel généraliste qui absorbe tous les métiers.

## 12. La décision humaine reste explicite

Mbàmbulaan peut :

- consolider ;
- alerter ;
- comparer ;
- recommander ;
- expliquer ;
- simuler.

Mais les décisions à fort impact doivent rester attribuées à des personnes ou instances clairement identifiées.

Toute recommandation automatisée doit être explicable, contestable et traçable.

## 13. La mesure doit distinguer activité, résultat, effet et impact

Nous ne confondons pas :

- activité réalisée ;
- livrable produit ;
- résultat obtenu ;
- effet sur les acteurs ;
- impact durable sur la filière.

Le produit doit rendre cette distinction visible pour éviter les reporting trompeurs.

## 14. Nous construisons pour la réplication

Une initiative utile ne doit pas rester isolée.

Mbàmbulaan doit permettre de comprendre :

- ce qui a fonctionné ;
- dans quel contexte ;
- avec quels acteurs ;
- avec quelles ressources ;
- avec quels résultats ;
- sous quelles conditions cela peut être reproduit.

La connaissance devient un actif stratégique de la plateforme.

## 15. Le produit standard prime sur le sur-mesure

Un besoin spécifique peut être accepté s'il :

- renforce une capacité générique ;
- peut servir à plusieurs acteurs ;
- respecte l'architecture ;
- ne fragilise pas le modèle économique ;
- reste configurable plutôt que codé pour un seul client.

Nous refusons les développements qui transforment Mbàmbulaan en logiciel propriétaire d'un commanditaire unique.

## 16. Le modèle économique doit être aligné avec la valeur créée

Le payeur n'est pas toujours l'utilisateur.

Nous devons identifier clairement :

- qui utilise ;
- qui bénéficie ;
- qui décide ;
- qui finance ;
- pourquoi il paie ;
- pourquoi il renouvelle.

La gratuité terrain peut être nécessaire, mais elle doit être financée par un modèle durable.

## 17. Aucun financeur ne doit capturer le produit

Un ministère, bailleur, investisseur ou grand client peut accélérer Mbàmbulaan, mais ne doit pas pouvoir imposer une orientation contraire à l'intérêt de l'écosystème et à la viabilité de l'entreprise.

Nous devons protéger :

- l'indépendance produit ;
- la propriété intellectuelle ;
- la capacité de réutilisation ;
- la diversification des revenus ;
- la gouvernance stratégique.

## 18. La sobriété précède la complexité

Nous privilégions :

- les parcours simples ;
- les objets essentiels ;
- les règles compréhensibles ;
- les coûts maîtrisés ;
- les technologies éprouvées ;
- l'automatisation progressive.

Nous ne devons pas ajouter de l'intelligence artificielle, de la blockchain ou des intégrations complexes sans preuve de valeur métier.

## 19. La roadmap est guidée par les preuves

Une phase n'est pas terminée parce que le calendrier l'indique.

Elle est terminée lorsque les preuves attendues existent :

- utilité ;
- adoption ;
- coordination ;
- qualité ;
- paiement ;
- réplication.

Le financement doit acheter de l'apprentissage avant d'acheter de la complexité.

## 20. Les acteurs vulnérables ne doivent pas supporter le coût du système

Les acteurs disposant du moins de pouvoir économique ou institutionnel ne doivent pas être ceux qui financent la majeure partie de l'infrastructure.

Les modèles tarifaires doivent protéger l'accès des acteurs terrain et éviter de renforcer les asymétries existantes.

## 21. La transparence a des limites légitimes

Tout ne doit pas être visible par tous.

Le produit doit équilibrer :

- transparence utile ;
- confidentialité ;
- intérêt collectif ;
- protection économique ;
- protection des données personnelles ;
- sécurité des acteurs.

Le partage doit être proportionné à la finalité.

## 22. Une fonctionnalité doit avoir un responsable et une métrique

Aucune fonctionnalité ne doit entrer dans la roadmap sans préciser :

- le problème traité ;
- l'acteur principal ;
- le bénéficiaire ;
- la décision améliorée ;
- le moteur propriétaire ;
- la métrique de succès ;
- le coût ;
- le responsable produit.

## 23. Les opérations font partie du produit

Au début, certaines étapes seront réalisées manuellement.

Ce n'est pas un échec si cela permet :

- d'apprendre ;
- de garantir la qualité ;
- de comprendre les exceptions ;
- de tester le modèle ;
- d'éviter une automatisation prématurée.

L'objectif est ensuite de transformer les opérations répétables en capacités produit.

## 24. Nous protégeons la continuité institutionnelle

Les données, décisions et apprentissages ne doivent pas disparaître lors d'un changement de personne, de programme, de bailleur ou d'administration.

Mbàmbulaan doit créer une mémoire durable et gouvernée de la transformation de la filière.

## 25. Nous arbitrons avec une grille constante

Pour chaque décision produit, nous demandons :

- est-ce que cela améliore la coordination ?
- est-ce que cela crée une vraie valeur métier ?
- qui utilise ?
- qui bénéficie ?
- qui décide ?
- qui paie ?
- pourquoi paierait-il durablement ?
- est-ce rentable pour Mbàmbulaan ?
- est-ce différenciant ?
- est-ce nécessaire maintenant ou plus tard ?

## 26. Signaux d'alerte

Une décision doit être challengée lorsqu'elle :

- ajoute un écran sans améliorer un flux ;
- crée une saisie sans retour de valeur ;
- dépend d'un seul client ;
- exige du sur-mesure non réutilisable ;
- collecte des données sans finalité claire ;
- centralise inutilement ;
- complexifie le produit avant validation ;
- remplace un système spécialisé sans raison ;
- exclut les acteurs terrain ;
- augmente les coûts sans revenu ou impact identifiable.

## 27. Principe directeur

> Mbàmbulaan doit rester utile au terrain, crédible pour les institutions, rentable pour l'entreprise et bénéfique pour l'ensemble de l'écosystème.
