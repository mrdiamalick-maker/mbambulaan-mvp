# 07 — MVP détaillé

## 1. Finalité

Le MVP de Mbàmbulaan doit prouver qu'une infrastructure numérique de coordination peut produire une valeur opérationnelle réelle dans la pêche artisanale sénégalaise.

Il ne doit pas chercher à couvrir toute la filière ni à reproduire un ERP complet.

Il doit démontrer qu'il est possible de :

- faire remonter une situation terrain ;
- la qualifier ;
- mobiliser les bons acteurs ;
- obtenir des engagements ;
- suivre les actions ;
- documenter les décisions ;
- rendre visible la progression ;
- produire des preuves d'utilité et de confiance.

## 2. Hypothèse centrale du MVP

> Si les acteurs d'un territoire disposent d'un espace simple pour signaler, qualifier, décider, s'engager et suivre l'exécution, alors la coordination devient plus rapide, plus traçable et plus fiable.

## 3. Objectifs du MVP

Le MVP doit valider cinq hypothèses :

### 3.1 Utilité opérationnelle

Les acteurs utilisent réellement le produit pour traiter des situations concrètes.

### 3.2 Capacité de coordination

Mbàmbulaan réduit les pertes d'information, les relances manuelles et les ambiguïtés de responsabilité.

### 3.3 Confiance

Les acteurs considèrent la chronologie, les preuves, les décisions et les engagements comme fiables.

### 3.4 Adoption institutionnelle

Une organisation coordinatrice ou publique accepte d'utiliser Mbàmbulaan dans ses processus.

### 3.5 Monétisation potentielle

Au moins un acteur disposant d'un budget reconnaît une valeur économique suffisante pour financer le service, un pilote ou un déploiement.

## 4. Périmètre métier recommandé

Le MVP doit se concentrer sur un seul flux de valeur :

```text
Signalement terrain
→ Qualification
→ Priorisation
→ Coordination
→ Engagements
→ Actions
→ Preuves
→ Clôture
```

Ce flux est suffisamment transversal pour démontrer la valeur de l'écosystème sans construire toute la plateforme.

## 5. Cas d'usage pilote

Le cas d'usage doit porter sur des situations :

- fréquentes ;
- concrètes ;
- impliquant plusieurs acteurs ;
- aujourd'hui mal suivies ;
- ayant un coût visible ;
- pouvant être résolues en quelques jours ou semaines.

Exemples pertinents :

- panne d'un équipement collectif ;
- indisponibilité de glace ;
- dysfonctionnement d'une infrastructure de débarquement ;
- problème sanitaire ou de salubrité ;
- besoin opérationnel partagé ;
- incident logistique ;
- action décidée mais non exécutée ;
- coordination d'une intervention institutionnelle.

## 6. Territoire pilote

Le MVP doit être déployé sur un territoire limité.

Critères de sélection :

- concentration d'acteurs ;
- fréquence des problèmes de coordination ;
- présence d'une organisation relais ;
- accessibilité terrain ;
- volonté locale ;
- possibilité de mesurer les résultats ;
- soutien institutionnel minimal.

Le territoire pilote ne doit pas être choisi uniquement pour sa visibilité politique.

## 7. Acteurs du MVP

### Acteur terrain

Exemples : pêcheur, mareyeur, transformateur, responsable de site.

Capacités :

- soumettre un signalement ;
- joindre une preuve ;
- suivre le traitement ;
- confirmer une action ;
- recevoir une notification.

### Coordinateur local

Capacités :

- recevoir les signalements ;
- qualifier les situations ;
- rattacher des acteurs ;
- créer un espace de coordination ;
- proposer des engagements ;
- assigner des actions ;
- relancer ;
- clôturer avec justification.

### Décideur ou autorité

Capacités :

- consulter les preuves ;
- prioriser ;
- prendre ou valider une décision ;
- désigner un responsable ;
- suivre les situations critiques.

### Partenaire d'exécution

Capacités :

- accepter un engagement ;
- exécuter une action ;
- signaler un blocage ;
- joindre une preuve ;
- confirmer la fin d'une intervention.

### Administrateur Mbàmbulaan

Capacités :

- configurer les organisations et territoires ;
- gérer les mandats ;
- assister les utilisateurs ;
- surveiller les incidents ;
- corriger les données avec traçabilité.

## 8. Parcours MVP principal

### Étape 1 — Signaler

L'acteur :

- choisit un type de situation ;
- décrit le problème ;
- indique le territoire ;
- ajoute éventuellement une photo ou un document ;
- soumet.

Résultat :

- accusé de réception ;
- identifiant de suivi ;
- statut visible.

### Étape 2 — Qualifier

Le coordinateur :

- vérifie les informations ;
- demande un complément si nécessaire ;
- évalue urgence et criticité ;
- rattache les signalements similaires ;
- identifie les acteurs concernés ;
- qualifie la situation.

### Étape 3 — Prioriser et décider

Le décideur ou coordinateur autorisé :

- examine la situation ;
- consulte les preuves ;
- confirme la priorité ;
- prend une décision ;
- désigne les responsables.

### Étape 4 — Coordonner

Le coordinateur :

- crée l'espace de coordination ;
- ajoute les participants ;
- définit les engagements ;
- fixe les échéances ;
- identifie les preuves attendues.

### Étape 5 — Exécuter

Les responsables :

- acceptent leurs engagements ;
- démarrent les actions ;
- déclarent les blocages ;
- mettent à jour l'état ;
- joignent les preuves.

### Étape 6 — Vérifier et clôturer

Le coordinateur ou validateur :

- vérifie les preuves ;
- confirme l'exécution ;
- documente le résultat ;
- clôture la situation ;
- communique le résultat au déclarant.

## 11. Hors périmètre du MVP

Le MVP ne doit pas inclure :

- marketplace ;
- paiement natif ;
- crédit ;
- assurance ;
- gestion complète des stocks ;
- traçabilité complète des produits de la mer ;
- gestion des licences ;
- contrôle maritime ;
- ERP des organisations ;
- intelligence artificielle prédictive ;
- scoring automatisé ;
- application citoyenne ouverte à tous ;
- API publique ;
- analytics avancés ;
- déploiement national ;
- gestion complète des financements ;
- gouvernance multi-ministérielle complexe.

## 13. Critères de succès produit

### Adoption

- nombre d'acteurs actifs ;
- taux d'activation ;
- taux de retour hebdomadaire ;
- nombre de signalements utiles ;
- proportion de profils cibles actifs.

### Coordination

- délai entre signalement et prise en charge ;
- délai de qualification ;
- délai de décision ;
- taux d'engagements acceptés ;
- taux d'actions réalisées à temps ;
- nombre de blocages résolus.

### Confiance

- taux d'objets avec preuve ;
- taux de décisions justifiées ;
- taux de clôtures contestées ;
- niveau de satisfaction des acteurs ;
- perception de la fiabilité.

### Utilité

- situations effectivement résolues ;
- coûts ou délais évités ;
- interventions mieux coordonnées ;
- taux de réutilisation du produit.

### Monétisation

- volonté de paiement ;
- pilote financé ;
- budget identifié ;
- coût par situation traitée ;
- coût de déploiement par territoire.

## 14. Seuils de validation proposés

À adapter au terrain, mais le MVP devrait viser :

- 30 à 50 utilisateurs actifs ;
- 3 à 5 organisations impliquées ;
- 20 à 40 situations réelles traitées ;
- au moins 60 % des situations qualifiées en moins de 48 heures ;
- au moins 60 % des actions réalisées ou justifiées ;
- au moins 70 % des utilisateurs clés déclarant une amélioration de coordination ;
- au moins une organisation prête à financer la suite.

## 15. Durée recommandée

### Préparation — 4 à 6 semaines

- sélection du territoire ;
- mobilisation des acteurs ;
- cartographie du processus actuel ;
- définition du protocole pilote ;
- configuration ;
- formation.

### Construction — 10 à 14 semaines

- produit ;
- tests ;
- intégrations minimales ;
- préparation opérationnelle.

### Pilote terrain — 12 à 16 semaines

- usage réel ;
- support ;
- collecte des mesures ;
- ajustements.

### Évaluation — 2 à 4 semaines

- résultats ;
- décision de poursuivre ;
- modèle économique ;
- plan de déploiement.

## 18. Dépendances critiques

- organisation relais crédible ;
- accès aux acteurs ;
- mandat clair du coordinateur ;
- disponibilité d'un décideur ;
- budget SMS et support ;
- connectivité minimale ;
- règles de confidentialité ;
- protocole de résolution des conflits ;
- capacité de support terrain.

## 19. Risques principaux

### Faible adoption

Réponse : co-conception, formation, animation terrain, parcours très courts.

### Absence de pouvoir décisionnel

Réponse : définir les mandats avant le pilote.

### Utilisation comme simple outil de remontée

Réponse : imposer le suivi jusqu'à l'engagement et la clôture.

### Données peu fiables

Réponse : sources, preuves, validation et audit.

### Trop grand périmètre

Réponse : un flux, un territoire, peu de rôles, peu de statuts.

### Dépendance au ministère

Réponse : conserver plusieurs voies de financement et partenaires.

### Charge de support trop élevée

Réponse : simplicité, formation de relais, processus standardisés.

## 20. Stratégie de déploiement

### Phase 1 — Accompagnement fort

Mbàmbulaan accompagne directement les utilisateurs.

### Phase 2 — Relais local

Un coordinateur local prend en charge l'animation quotidienne.

### Phase 3 — Standardisation

Les pratiques validées deviennent un kit de déploiement.

### Phase 4 — Réplication

Le modèle est étendu à un second territoire comparable.

Le kit doit inclure :

- supports de formation ;

- modèle de rapport pilote ;
- règles d'escalade.

## 22. Décisions go / no-go

À la fin du pilote :

### Go

Si :

- l'usage est réel ;
- la coordination s'améliore ;
- les acteurs clés restent engagés ;
- les coûts sont soutenables ;
- un financeur est identifié.

### Pivot

Si :

- un segment utilise fortement une partie du flux ;
- la valeur existe mais sur un cas d'usage plus étroit ;
- le payeur réel diffère de l'hypothèse initiale.

### Stop

Si :

- le produit n'est pas utilisé malgré l'accompagnement ;
- aucun acteur n'assume la coordination ;
- les situations ne peuvent pas être résolues ;
- aucun modèle de financement crédible n'émerge.

## 23. Définition de terminé du MVP

Le MVP est terminé lorsqu'il est possible de démontrer, preuves à l'appui :

- qu'une situation réelle a été signalée ;
- qu'elle a été qualifiée ;
- qu'une décision a été prise ;
- que plusieurs acteurs se sont engagés ;
- que les actions ont été suivies ;
- que le résultat a été vérifié ;
- que les utilisateurs perçoivent une amélioration ;
- qu'un acteur est prêt à financer la suite.

## 24. Principe directeur

> Le MVP de Mbàmbulaan ne doit pas prouver que nous savons construire une application. Il doit prouver que nous savons améliorer une coordination réelle et créer une valeur suffisamment forte pour être financée.
