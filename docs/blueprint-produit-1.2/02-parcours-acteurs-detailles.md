# 02 — Parcours acteurs détaillés

## 1. Finalité

Ce document décrit les parcours clés de Mbàmbulaan par profil d'acteur.

Il ne s'agit pas de lister des écrans de manière abstraite, mais de comprendre :

- ce que chaque acteur cherche à accomplir ;
- ce qui déclenche son usage ;
- les décisions qu'il doit prendre ;
- les informations dont il a besoin ;
- les blocages possibles ;
- la valeur qu'il retire du produit ;
- la contribution de son parcours à la coordination globale.

## 2. Principes de conception des parcours

Les parcours doivent être :

- courts ;
- compréhensibles ;
- orientés action ;
- adaptés au mobile ;
- compatibles avec une connectivité instable ;
- contextualisés selon le rôle ;
- conçus pour éviter les doublons et les responsabilités floues ;
- cohérents avec le lexique métier.

## 3. Acteurs couverts dans le MVP

Les parcours principaux concernent :

1. l'acteur terrain ;
2. le coordinateur local ;
3. le décideur ou autorité ;
4. le partenaire d'exécution ;
5. l'administrateur Mbàmbulaan.

## 4. Parcours 1 — Acteur terrain

### 4.1 Objectif principal

Permettre à un acteur de terrain de faire remonter un problème, un besoin ou une situation et d'en suivre la prise en charge.

### 4.2 Déclencheurs

Le parcours commence lorsqu'il :

- constate une panne ;
- observe un problème de salubrité ;
- identifie un besoin collectif ;
- subit un blocage opérationnel ;
- veut attirer l'attention sur une situation urgente.

### 4.3 Étapes du parcours

#### Étape 1 — Accès

L'acteur :

- ouvre Mbàmbulaan ;
- saisit son numéro de téléphone ;
- reçoit un code ;
- accède à son espace.

#### Étape 2 — Choix de l'action

L'acteur voit des actions simples :

- Signaler un problème ;
- Signaler un besoin ;
- Suivre mes remontées.

#### Étape 3 — Saisie

L'acteur renseigne :

- le type de situation ;
- une description courte ;
- le lieu ou territoire ;
- une photo ou un document si utile ;
- le niveau d'urgence perçu.

#### Étape 4 — Confirmation

Le système :

- affiche un récapitulatif ;
- confirme l'envoi ;
- génère un identifiant de suivi ;
- indique la prochaine étape.

#### Étape 5 — Suivi

L'acteur consulte :

- le statut ;
- la date de prise en charge ;
- la personne ou structure en charge ;
- les demandes de complément ;
- la décision ou action engagée ;
- la clôture.

#### Étape 6 — Confirmation du résultat

Lorsque la situation est résolue, l'acteur peut :

- confirmer que le résultat est visible ;
- signaler que le problème persiste ;
- ajouter un commentaire ;
- joindre une photo si nécessaire.

### 4.4 Informations prioritaires

L'acteur doit toujours comprendre :

- ce qui a été reçu ;
- qui s'en occupe ;
- ce qui va se passer ensuite ;
- où en est la situation ;
- pourquoi elle a été clôturée.

### 4.5 Frictions à éviter

- formulaire trop long ;
- vocabulaire administratif ;
- absence de retour après l'envoi ;
- statuts incompréhensibles ;
- demande de document non justifiée ;
- impossibilité de corriger une erreur ;
- doublons non détectés.

### 4.6 Valeur créée

Pour l'acteur :

- sa remontée n'est plus perdue ;
- il voit la prise en charge ;
- il évite des relances multiples ;
- il comprend les responsabilités ;
- il peut confirmer le résultat.

## 5. Parcours 2 — Coordinateur local

### 5.1 Objectif principal

Transformer les remontées en situations actionnables et organiser la coordination entre acteurs.

### 5.2 Déclencheurs

Le parcours commence lorsqu'un :

- nouveau signalement arrive ;
- complément est reçu ;
- blocage est déclaré ;
- engagement arrive à échéance ;
- décideur demande un suivi.

### 5.3 Étapes du parcours

#### Étape 1 — Vue de travail

Le coordinateur voit :

- les nouveaux signalements ;
- les situations urgentes ;
- les actions bloquées ;
- les validations attendues ;
- les échéances proches.

#### Étape 2 — Analyse

Il ouvre une remontée et vérifie :

- la description ;
- le lieu ;
- l'auteur ;
- les pièces jointes ;
- les remontées similaires ;
- les acteurs concernés.

#### Étape 3 — Qualification

Il renseigne :

- la nature de la situation ;
- l'urgence ;
- la criticité ;
- le périmètre ;
- l'orientation ;
- la justification.

#### Étape 4 — Demande de complément

Si nécessaire, il demande :

- une précision ;
- une photo ;
- un document ;
- une validation locale ;
- l'identification d'un lieu ou acteur.

#### Étape 5 — Création de la situation

Il transforme ou regroupe les remontées en une situation structurée.

#### Étape 6 — Préparation de la décision

Il :

- identifie les options ;
- propose une priorité ;
- mobilise les acteurs concernés ;
- prépare les éléments nécessaires à la décision.

#### Étape 7 — Mise en coordination

Il :

- crée l'espace de coordination ;
- ajoute les participants ;
- formalise les engagements ;
- crée les actions ;
- définit les échéances ;
- indique les confirmations attendues.

#### Étape 8 — Suivi

Il suit :

- les actions en retard ;
- les blocages ;
- les engagements non acceptés ;
- les validations en attente ;
- les mises à jour terrain.

#### Étape 9 — Clôture

Il :

- vérifie que les actions sont terminées ;
- contrôle les éléments de confirmation ;
- recueille la validation si nécessaire ;
- documente le résultat ;
- clôture avec un motif clair.

### 5.4 Informations prioritaires

Le coordinateur doit toujours voir :

- ce qui nécessite son attention ;
- qui est responsable ;
- la prochaine action ;
- les échéances ;
- les blocages ;
- les décisions attendues.

### 5.5 Frictions à éviter

- surcharge d'informations ;
- absence de priorité ;
- actions sans responsable ;
- multiplication des commentaires libres ;
- manque de visibilité sur les délais ;
- difficulté à regrouper les remontées ;
- clôture sans validation.

### 5.6 Valeur créée

Pour le coordinateur :

- vue centralisée ;
- responsabilités explicites ;
- moins de relances dispersées ;
- meilleure traçabilité ;
- capacité à rendre compte ;
- meilleure priorisation.

## 6. Parcours 3 — Décideur ou autorité

### 6.1 Objectif principal

Prendre des décisions rapides, justifiées et suivies.

### 6.2 Déclencheurs

Le parcours commence lorsqu'une situation :

- dépasse le mandat du coordinateur ;
- nécessite un arbitrage ;
- implique un budget ;
- présente une criticité élevée ;
- concerne plusieurs organisations.

### 6.3 Étapes du parcours

#### Étape 1 — Vue synthèse

Le décideur voit :

- les situations à décider ;
- les situations critiques ;
- les décisions en retard ;
- les impacts potentiels ;
- les territoires concernés.

#### Étape 2 — Lecture de la situation

Il consulte :

- le résumé ;
- les faits essentiels ;
- l'urgence ;
- la criticité ;
- les acteurs concernés ;
- les options proposées ;
- les éléments disponibles.

#### Étape 3 — Arbitrage

Il peut :

- approuver une option ;
- demander un complément ;
- modifier la priorité ;
- désigner un responsable ;
- refuser avec justification ;
- reporter la décision.

#### Étape 4 — Formalisation

La décision précise :

- le décideur ;
- le contenu ;
- la justification ;
- les responsables ;
- les échéances ;
- les conséquences.

#### Étape 5 — Suivi

Le décideur voit :

- si la décision a été transformée en actions ;
- les engagements pris ;
- les blocages ;
- le résultat final ;
- les écarts éventuels.

### 6.4 Informations prioritaires

- ce qui nécessite une décision ;
- les conséquences d'une non-décision ;
- les options ;
- les responsables ;
- les délais ;
- le résultat attendu.

### 6.5 Frictions à éviter

- détail excessif ;
- absence de synthèse ;
- demandes non priorisées ;
- arbitrages sans conséquence visible ;
- décisions non suivies ;
- confusion entre information et action.

### 6.6 Valeur créée

Pour le décideur :

- information consolidée ;
- arbitrages plus rapides ;
- justification conservée ;
- suivi de l'exécution ;
- meilleure redevabilité ;
- vue territoriale fiable.

## 7. Parcours 4 — Partenaire d'exécution

### 7.1 Objectif principal

Recevoir une mission claire, l'accepter, l'exécuter et confirmer le résultat.

### 7.2 Déclencheurs

Le parcours commence lorsqu'il :

- reçoit une action ;
- reçoit une proposition d'engagement ;
- doit intervenir sur un site ;
- doit fournir un document ou une validation.

### 7.3 Étapes du parcours

#### Étape 1 — Notification

Le partenaire reçoit :

- l'objet de la demande ;
- le lieu ;
- l'échéance ;
- le responsable de coordination ;
- le niveau de priorité.

#### Étape 2 — Consultation

Il consulte :

- le contexte ;
- le résultat attendu ;
- les contraintes ;
- les acteurs concernés ;
- la confirmation attendue.

#### Étape 3 — Acceptation

Il peut :

- accepter ;
- proposer une autre échéance ;
- refuser avec motif ;
- demander une clarification.

#### Étape 4 — Exécution

Il met à jour :

- démarré ;
- en cours ;
- bloqué ;
- terminé.

#### Étape 5 — Confirmation

Il ajoute selon le contexte :

- une photo ;
- un document ;
- un compte rendu ;
- une confirmation d'intervention ;
- une demande de validation.

#### Étape 6 — Validation

Il reçoit :

- la validation ;
- une demande de complément ;
- un rejet motivé ;
- la clôture.

### 7.4 Frictions à éviter

- mission peu claire ;
- absence d'échéance ;
- confirmation demandée trop tard ;
- absence de canal de blocage ;
- impossibilité de proposer une nouvelle date ;
- validation lente.

### 7.5 Valeur créée

Pour le partenaire :

- demandes plus claires ;
- historique fiable ;
- réduction des malentendus ;
- meilleure coordination avec le terrain ;
- reconnaissance de l'intervention réalisée.

## 8. Parcours 5 — Administrateur Mbàmbulaan

### 8.1 Objectif principal

Configurer et sécuriser l'usage du produit sans dépendre systématiquement de l'équipe technique.

### 8.2 Déclencheurs

- onboarding d'une organisation ;
- création d'un territoire ;
- ajout d'un acteur ;
- attribution d'un mandat ;
- correction de données ;
- incident de synchronisation ;
- demande de support.

### 8.3 Étapes du parcours

#### Étape 1 — Configuration

L'administrateur gère :

- territoires ;
- organisations ;
- rôles ;
- mandats ;
- types de signalement ;
- niveaux de priorité ;
- référentiels.

#### Étape 2 — Onboarding

Il :

- crée ou invite les acteurs ;
- les rattache aux organisations ;
- attribue les rôles ;
- vérifie les coordonnées ;
- suit l'activation.

#### Étape 3 — Support

Il recherche :

- un utilisateur ;
- une situation ;
- une erreur ;
- une synchronisation échouée ;
- un mandat expiré.

#### Étape 4 — Correction

Il peut corriger une donnée avec :

- motif ;
- auteur ;
- date ;
- ancienne valeur ;
- nouvelle valeur.

#### Étape 5 — Surveillance

Il suit :

- erreurs ;
- accès refusés ;
- doublons ;
- actions en échec ;
- notifications non délivrées.

### 8.4 Frictions à éviter

- actions irréversibles ;
- manque d'historique ;
- trop de droits ;
- correction silencieuse ;
- dépendance à des scripts ;
- absence d'alerte.

### 8.5 Valeur créée

Pour Mbàmbulaan :

- déploiement plus rapide ;
- support plus efficace ;
- meilleure qualité des données ;
- réduction des interventions techniques ;
- sécurité renforcée.

## 9. Parcours transversal — Demande de complément

### Objectif

Permettre à un acteur de compléter une information sans recréer une nouvelle remontée.

### Étapes

1. Le coordinateur formule une demande précise.
2. L'acteur reçoit une notification claire.
3. Il ajoute l'information demandée.
4. Le système rattache le complément à la situation existante.
5. Le coordinateur reprend l'analyse.

### Règles

- préciser ce qui manque ;
- expliquer pourquoi ;
- éviter les demandes vagues ;
- conserver l'historique ;
- notifier la réception du complément.

## 10. Parcours transversal — Blocage

### Objectif

Rendre visible ce qui empêche la progression.

### Étapes

1. Un acteur déclare un blocage.
2. Il précise la cause et l'impact.
3. Un responsable de résolution est désigné.
4. Une action de résolution est créée.
5. Le blocage est suivi jusqu'à levée.

### Règles

- un blocage ne doit pas être un simple commentaire ;
- il doit avoir un propriétaire ;
- il doit avoir une prochaine action ;
- il doit être visible dans les vues de pilotage.

## 11. Parcours transversal — Clôture

### Objectif

Clore une situation de manière compréhensible et vérifiable.

### Étapes

1. Les actions sont terminées.
2. Les confirmations attendues sont disponibles.
3. Le résultat est vérifié.
4. Le déclarant ou bénéficiaire peut donner son retour.
5. Le coordinateur choisit un motif de clôture.
6. La situation est clôturée.

### Motifs de clôture possibles

- résolu ;
- partiellement résolu ;
- transféré ;
- non fondé ;
- abandonné ;
- impossible à traiter ;
- doublon.

## 12. États communs à tous les parcours

Chaque écran ou objet doit prévoir :

- chargement ;
- vide ;
- succès ;
- erreur ;
- hors ligne ;
- synchronisation en attente ;
- conflit ;
- accès refusé ;
- information incomplète ;
- action impossible.

## 13. Notifications essentielles

Les notifications doivent être limitées aux moments utiles :

- remontée reçue ;
- prise en charge ;
- complément demandé ;
- décision prise ;
- action assignée ;
- engagement à accepter ;
- échéance proche ;
- blocage déclaré ;
- validation demandée ;
- situation clôturée.

## 14. Critères de qualité d'un parcours

Un parcours est acceptable s'il permet de répondre clairement à :

- qui agit ?
- pourquoi ?
- sur quel objet ?
- avec quelle information ?
- quelle décision est attendue ?
- quelle est la prochaine étape ?
- qui bénéficie ?
- comment le résultat est-il confirmé ?

## 15. Priorités MVP

### À implémenter en premier

- acteur terrain : signaler et suivre ;
- coordinateur : qualifier et coordonner ;
- décideur : décider ;
- partenaire : accepter et exécuter ;
- administrateur : configurer les accès essentiels.

### À reporter

- personnalisation avancée par organisation ;
- parcours de financement complet ;
- gestion commerciale ;
- analytics avancés ;
- automatisations complexes ;
- parcours publics ouverts.

## 16. Principe directeur

> Chaque parcours de Mbàmbulaan doit réduire l'incertitude, rendre la responsabilité visible et faire progresser une situation jusqu'à une issue comprise et confirmée par les acteurs concernés.
